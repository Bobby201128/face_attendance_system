# 人脸识别签到系统 - 优化需求文档

## 一、项目背景

当前系统是一个基于人脸识别的上班打卡签到模式，包含PC端和移动端。本文档详细描述系统的优化需求，包括环境系统、人员库系统、类型编辑系统以及人脸录入方式的调整。

---

## 二、系统架构概览

### 2.1 当前架构
- **PC端**: PyQt5桌面应用，负责人脸识别签到、显示摄像头画面、统计数据
- **移动端**: Web管理界面，负责人员管理、签到记录查询、系统设置
- **数据库**: SQLite，存储人员信息、签到记录、系统配置
- **API服务**: Flask提供RESTful接口

### 2.2 优化后的架构
- **环境配置层**: 新增环境系统，支持多环境配置
- **人员管理层**: 扩展人员库，支持自定义类型和属性
- **人脸识别层**: 调整为人脸信息由手机端录入，PC端仅负责识别
- **签到逻辑层**: 可配置的签到规则，与环境绑定

## 三、详细需求说明

### 3.1 环境系统

#### 3.1.1 功能概述
新增环境系统，允许管理员在手机端配置多个环境，PC端可以选择当前使用的环境。每个环境可以独立配置签到逻辑。

#### 3.1.2 环境配置项
每个环境包含以下配置：

| 配置项 | 说明 | 数据类型 | 默认值 |
|--------|------|----------|--------|
| 环境名称 | 环境的唯一标识 | 字符串 | 必填 |
| 环境描述 | 环境说明 | 字符串 | 可选 |
| 签到模式 | 签到/签退/双向 | 枚举 | 双向 |
| 上班时间 | 签到开始时间 | 时间 | 09:00 |
| 下班时间 | 签退开始时间 | 时间 | 18:00 |
| 迟到阈值 | 超过此时长算迟到 | 分钟 | 15 |
| 早退阈值 | 提前此时长算早退 | 分钟 | 15 |
| 识别阈值 | 人脸识别置信度 | 浮点数 | 0.6 |
| 确认帧数 | 连续识别帧数 | 整数 | 3 |
| 签到冷却 | 同一人签到间隔 | 秒 | 60 |
| 是否启用 | 环境是否可用 | 布尔 | 是 |

#### 3.1.3 环境管理功能（手机端）
- **创建环境**: 新增环境配置
- **编辑环境**: 修改已有环境配置
- **删除环境**: 删除不再使用的环境（需确认无关联数据）
- **复制环境**: 基于现有环境创建新环境
- **环境列表**: 展示所有环境，支持搜索和筛选
- **设置默认环境**: 指定系统默认使用的环境

#### 3.1.4 环境切换功能（PC端）
- **环境选择器**: 下拉框展示可用环境
- **环境信息展示**: 显示当前环境的关键配置
- **切换确认**: 切换环境时给出确认提示
- **实时生效**: 切换后签到逻辑立即更新

#### 3.1.5 数据库设计
```sql
-- 环境配置表
CREATE TABLE IF NOT EXISTS environments (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL UNIQUE,
    description TEXT DEFAULT '',
    sign_mode TEXT DEFAULT 'both',
    work_start TEXT DEFAULT '09:00',
    work_end TEXT DEFAULT '18:00',
    late_grace INTEGER DEFAULT 15,
    early_leave_grace INTEGER DEFAULT 15,
    recognition_threshold REAL DEFAULT 0.6,
    confirm_frames INTEGER DEFAULT 3,
    sign_cooldown INTEGER DEFAULT 60,
    is_active INTEGER DEFAULT 1,
    is_default INTEGER DEFAULT 0,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 签到记录表新增环境ID字段
ALTER TABLE attendance ADD COLUMN environment_id INTEGER;
```

### 3.3 类型编辑系统

#### 3.3.1 功能概述
新增类型配置系统，允许管理员自定义配置组织架构类型（如集团、部门、小组等），每个大类下可配置多个选项。添加人员时使用二级选择框进行选择。

#### 3.3.2 类型层级结构
```
类型系统
├── 大类1 (例如: 集团)
│   ├── 选项1 (例如: 总部)
│   ├── 选项2 (例如: 上海分公司)
│   └── 选项3 (例如: 北京分公司)
├── 大类2 (例如: 部门)
│   ├── 选项1 (例如: 技术部)
│   ├── 选项2 (例如: 市场部)
│   └── 选项3 (例如: 人事部)
└── 大类3 (例如: 岗位)
    ├── 选项1 (例如: 经理)
    ├── 选项2 (例如: 工程师)
    └── 选项3 (例如: 实习生)
```

#### 3.3.3 类型管理功能（手机端）
- **添加大类**: 新增类型大类（如集团、部门）
- **编辑大类**: 修改大类名称和描述
- **删除大类**: 删除不再使用的大类（需确认无关联数据）
- **添加选项**: 在指定大类下添加选项
- **编辑选项**: 修改选项名称
- **删除选项**: 删除大类下的选项
- **排序功能**: 支持大类和选项的排序调整
- **启用/禁用**: 可临时禁用某个大类或选项

#### 3.3.4 二级选择框（添加/编辑人员时）
- **第一级**: 选择大类（如部门）
- **第二级**: 选择该大类下的具体选项（如技术部）
- **多类型支持**: 一个人员可以同时配置多个类型属性
- **必填/可选**: 可配置哪些类型是必填项

#### 3.3.5 数据库设计
```sql
-- 类型大类表
CREATE TABLE IF NOT EXISTS type_categories (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL UNIQUE,
    description TEXT DEFAULT '',
    sort_order INTEGER DEFAULT 0,
    is_required INTEGER DEFAULT 0,
    is_active INTEGER DEFAULT 1,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 类型选项表
CREATE TABLE IF NOT EXISTS type_options (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    category_id INTEGER NOT NULL,
    name TEXT NOT NULL,
    sort_order INTEGER DEFAULT 0,
    is_active INTEGER DEFAULT 1,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (category_id) REFERENCES type_categories(id)
);

-- 人员类型关联表
CREATE TABLE IF NOT EXISTS person_type_links (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    person_id INTEGER NOT NULL,
    category_id INTEGER NOT NULL,
    option_id INTEGER NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (person_id) REFERENCES persons(id),
    FOREIGN KEY (category_id) REFERENCES type_categories(id),
    FOREIGN KEY (option_id) REFERENCES type_options(id),
    UNIQUE(person_id, category_id)
);
```

### 3.4 人脸录入方式调整

#### 3.4.1 功能概述
修改系统架构，人脸信息的录入统一由手机端完成，PC端仅负责人脸检测和识别签到，不再支持人脸录入功能。

#### 3.4.2 手机端人脸录入流程
1. **选择人员**: 在人员列表中选择需要录入人脸的人员
2. **启动摄像头**: 调用手机摄像头进行拍照
3. **人脸检测**: 实时检测画面中的人脸
4. **拍照确认**: 用户确认拍摄的照片
5. **上传处理**: 将照片上传到服务器
6. **提取特征**: 服务器端提取人脸特征编码
7. **保存数据**: 将人脸编码保存到数据库
8. **结果反馈**: 告知用户录入成功或失败

#### 3.4.3 PC端功能调整
- **移除录入功能**: 移除PC端的人脸录入界面和相关代码
- **保留识别功能**: 保留人脸识别签到功能
- **信息展示**: 显示当前已录入的人员列表（只读）

#### 3.4.4 API接口调整
- 人脸录入接口保持不变
- 确保API可以正确处理手机端上传的图片
- 添加图片质量检测功能

#### 3.4.5 人脸录入优化建议
- **多张照片录入**: 支持同一人录入多张照片，提高识别准确率
- **质量检测**: 检测照片清晰度、光照、角度等
- **活体检测**: 简单的活体检测（如眨眼提示）
- **录入指引**: 引导用户拍摄合格的人脸照片
